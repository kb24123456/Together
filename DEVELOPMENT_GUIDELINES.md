# DEVELOPMENT_GUIDELINES.md

## 1. 目标
- 本规范服务于项目「一起 / Toghter」的 iPhone 端开发，主要供 Codex Mac 客户端执行。
- 优先级：先把结构搭稳，再逐步开发页面；严格核心边界，灵活实现细节。
- 所有实现必须简单、清晰、可维护、可被 AI 理解，避免过度抽象和炫技。

## 2. 项目边界
- 平台：iPhone only。
- 技术：SwiftUI 为主；仅在 SwiftUI 无法合理实现时，允许局部使用 UIKit。
- 环境：Xcode 26.2，Swift，iOS 18+。
- 产品边界：禁止演化为泛 Todo、泛社交、多人协作工具。
- 关系边界：仅支持一对一绑定。解绑后，旧双人关系数据不得迁移给新绑定对象；与新对象绑定必须从零开始。

## 3. 当前后端与能力策略
- 长期方案：CloudKit。
- 当前阶段：因未支付 Apple Developer 年费，先以本地数据 + mock 登录 + 可替换服务层实现。
- 登录：当前使用 mock Sign in with Apple 流程；接口和状态机必须按未来真实接入方式设计。
- 邀请与绑定：允许先实现“真实业务流的本地模拟版”，即完整走邀请/接受/绑定流程，但底层先用本地持久化或 mock service。
- 通知：首版支持本地提醒通知；推送通知接口先协议化预留。

## 4. 离线能力要求
- 必须支持离线浏览已保存的数据。
- 必须支持离线创建、编辑、完成事项与决策草稿。
- 必须支持离线查看纪念日与本地提醒。
- 必须支持网络恢复后的状态同步预留点，但当前本地版可先不做真正冲突合并。
- 首版不要求复杂多端冲突解决；但数据层设计不得阻断后续演进。

## 5. 架构原则
- 架构由 Codex 选择，但必须满足：
  - 业务逻辑与 View 分离。
  - 状态机不得直接散落在页面中。
  - 服务层先协议化，再实现具体服务。
  - 数据模型、状态流转、页面路由必须清晰可追踪。
- 优先选择适合独立开发 + AI 协作 + 后续扩展的稳定方案。
- 未经明确收益证明，不要引入复杂 Clean 分层或过重框架。

## 6. 工程组织
- 推荐至少包含：
  - App
  - Features
  - Shared
  - Models
  - Services
  - ViewModels
  - Components
  - Resources
  - Previews
  - Docs
- 每个页面必须拆分为主视图 + 子组件，避免单文件过大。
- 单个 ViewModel 不应承担多个页面或多个业务域。
- 所有 Design Tokens、常量、枚举、文案键值应集中管理，禁止 magic number 散落页面。

## 7. UI 开发原则
- 当前只写开发协作层规范，不固化完整视觉规范。
- 优先复用组件；非必要不允许页面内随意定义临时样式。
- 优先使用 iOS 26 原生液态玻璃/原生系统组件能力，但不得为了炫技牺牲可读性与性能。
- 页面必须支持 SwiftUI Preview 和 mock data 预览。
- 必须从开发阶段考虑深色模式。
- 动效要求：优先使用 Apple 原生 API，丝滑、克制、保证帧率，禁止无意义动画堆砌。
- 视觉方向约束：黑/白/粉为主，现代简洁，轻生活感，禁止廉价情侣风、过度粉色、过度装饰。

## 8. 核心产品规则
- 请求和待办本质上是同一类“事项”；事项卡片统一放在首页流中。
- 决策仅存在于“决策”页面；仅部分满足条件的决策可一键生成对应事项。
- 事项状态、决策状态、优先级档位不要在规范里写死为不可变常量，必须可扩展。
- 默认可以先实现一套首版状态与优先级，但代码层应支持后续扩展和用户自定义。
- 首页、决策、纪念日、我 为当前一级导航建议；若未来调整，必须先更新文档再改代码。

## 9. 默认首版建议值（可配置，不写死）
- 事项默认优先级：普通 / 重要 / 紧急。
- “无法完成”默认预设原因建议：
  - 这会儿没法处理
  - 时间来不及
  - 我不太适合做这件事
  - 需要你来决定
  - 晚点再说
- 决策默认三态语义建议：赞同 / 再想想 / 不考虑。

## 10. 页面开发顺序
- 先数据模型与状态机。
- 再服务层协议与本地实现。
- 再首页交互骨架。
- 再事项创建/确认主链路。
- 再决策页。
- 再纪念日页。
- 再我的页面与绑定流程。
- 最后统一视觉精修与动效。

## 11. AI 执行规范
- 复杂任务必须先输出：目标、影响范围、实现方案、风险点，再开始修改。
- 简单任务可直接执行，但完成后必须总结影响范围。
- 不允许跳过文档直接写页面。
- 不允许擅自新增功能、修改产品边界、改变数据规则。
- 不允许跨模块大改；必须小步提交、逐步验证。
- 每次改动优先保证可运行，再逐步优化。
- 所有不确定处必须显式标记为 Assumptions / Open Questions。

## 12. 代码风格
- 命名优先业务语义，避免泛化成 task manager 语言。
- enum 优先于魔法字符串；状态流转优先显式建模。
- 复杂逻辑优先拆到 ViewModel / Service / Reducer 风格对象中，不留在 View body。
- View 层只负责渲染、事件分发、轻量 UI 状态。
- Service 层负责数据访问、通知、登录、绑定等能力。
- 协议命名、Mock 命名、Preview 数据命名必须统一且可搜索。

## 13. 依赖与第三方库
- 必要时可使用稳定第三方库，但必须先说明用途、替代方案、引入成本。
- 非必要不引入第三方状态管理、网络、图片或动画大库。
- 优先使用 Apple 原生能力与标准库。

## 14. 测试与质量
- 不要求首版一次性铺满测试，但关键业务逻辑必须可测。
- 至少保证：
  - 状态机逻辑可单测。
  - 数据模型转换可单测。
  - 关键 ViewModel 行为可单测。
- UI 测试可后置，但主流程至少应可通过 Preview + mock data 快速回归。

## 15. 文档要求
- 保留并持续维护 AGENTS.md。
- 是否新增其他 docs 由 Codex 判断，但复杂模块至少应补充：架构说明、数据模型说明、状态机说明。
- 文档应简洁，避免无效冗长，优先可执行性。

## 16. Git 与提交
- 推荐提交类型：feat / fix / refactor / docs / chore。
- 每次提交只解决一个清晰问题，避免“大而全”。
- 提交前必须保证项目可编译、关键预览可运行。

## 17. 禁止事项
- 禁止把首页做成普通任务列表工具。
- 禁止把绑定关系做成多人或开放社交。
- 禁止为了快速实现而把核心状态机硬编码在单个页面中。
- 禁止大范围复制粘贴组件而不做抽象。
- 禁止为了视觉效果牺牲性能与可维护性。
