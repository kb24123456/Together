# DEVELOPMENT_GUIDELINES.md

## 1. 目标
- 本规范服务于项目「一起 / Together」的 iPhone 端开发。
- 当前最高优先级：把单人模式做成真实可用的 Todo 效率工具。
- 所有实现必须简单、清晰、可维护、可被 AI 理解。

## 2. 当前产品边界

### 事实
- V1 主轴已经从“双人协作优先”切换为“单人 Todo 优先”。
- 当前 MVP 主要服务个人工作 Todo、当前项目推进、日常待办管理。
- 当前首页静态 UI 可继续沿用，并作为单人模式 Today 首页的视觉基础。

### 开发结论
- 当前首版优先建设 `Today / 清单 / 项目 / 日历 / 我`。
- 双人模式为 V2 差异化扩展，不是当前主链路。
- 决策、纪念日降级为后续附加模块。
- 多人模式仅做架构预留，不做当前实现承诺。

## 3. 技术约束
- 平台：iPhone only。
- 技术：SwiftUI 为主；仅在 SwiftUI 无法合理实现时，局部使用 UIKit。
- 环境：Xcode 26.2，Swift，iOS 18+。
- 当前阶段：本地数据 + mock 登录 + 可替换服务层。
- 长期方案：CloudKit；当前实现不得阻断未来接入。
- 页面必须支持 SwiftUI Preview 和 mock data。
- 开发时就考虑深色模式、Reduce Motion 和安全区域。

## 4. 架构原则
- 业务逻辑与 View 分离。
- 状态机不得直接散落在 View 中。
- 服务层先协议化，再实现具体服务。
- 数据模型、状态流转、页面路由必须清晰可追踪。
- 新增模型应优先面向 `Task / List / Project / Calendar / Space` 抽象。
- 当前代码里旧的 `PairSpace / relationshipID / Invite` 命名属于历史包袱；新开发不要继续扩散这些语义。

## 5. 空间与数据策略
- 产品层当前默认只暴露单人工作空间，不对用户强调复杂 Space 概念。
- 技术层允许预留：
  - `SingleSpace`
  - `PairSpace`
  - `MultiSpace`
- V1 可以只让 `SingleSpace` 真正跑通。
- V2 再把双人协作挂到 `PairSpace` 上，而不是反过来把单人模式当试用态。

## 6. 当前 MVP 重点能力
- Task：创建、编辑、完成、重新打开、提醒、筛选、排序。
- List：系统清单与自定义清单。
- Project：项目归档、状态切换、项目内任务聚合。
- Calendar：周视图、月视图、按日期查看任务。
- Profile：通知、主题、默认视图、未来双人模式入口。

## 7. UI 与动效要求
- 当前首页 UI 不推翻重做，后续单人模式优先在现有骨架上深化。
- 原生组件优先采用 iOS 26 液态玻璃原生组件；优先使用 `glassEffect`、`GlassEffectContainer`、原生 glass button style，避免自制假玻璃。
- 优先使用 Apple 原生动画 API。
- 动效必须克制、自然、层级清晰、帧率稳定。
- 动效是产品亮点，不能放到开发末尾再补。
- 重点覆盖：
  - 任务完成与状态切换
  - 首页列表与卡片层级变化
  - 周视图 / 月视图切换
  - 主按钮展开与收起
  - 详情展开 / 收起
  - 筛选 / 排序 / 分组切换
- 禁止为了动效牺牲响应速度、可读性或安全区域。

## 8. 工程组织建议
- `App/`：App 入口、路由、会话、依赖容器
- `Domain/`：实体、枚举、协议、状态机
- `Services/`：repository、本地存储、通知、认证
- `Features/`：Today、Lists、Projects、Calendar、Profile、扩展模块
- `Shared/`：设计 token、基础组件、动画工具
- `PreviewContent/`：mock data、preview fixture

## 9. 默认开发顺序
1. 文档与边界统一
2. Task / List / Project / Space 模型与状态机
3. 服务协议与本地 mock / 持久化
4. Today 首页主链路
5. 清单与项目
6. 日历与时间视图
7. 创建入口、详情页、搜索筛选
8. 设置与提醒
9. 双人扩展与附加模块

## 10. AI 执行规范
- 复杂任务先给：目标、影响范围、方案、风险。
- 未读文档不要直接写页面。
- 优先更新现有文件；仅在缺失关键文档时新增文件。
- 每次改动先保证可运行，再做精修。
- 所有不确定处必须显式标记为 `Open Questions`。

## 11. 测试与验证
- 状态机逻辑必须可单测。
- 数据模型转换与 repository 行为必须可单测。
- 关键 ViewModel 行为必须可单测。
- 首页 / 日历 / 项目关键页面必须有 Preview + mock data。
- 只要涉及动效状态切换，就至少做一次真机构建或 Preview 回归。

## 12. 禁止事项
- 禁止继续把绑定流、邀请流、双人决策流当首版主目标。
- 禁止把当前产品重新做成情侣运营或关系运营工具。
- 禁止把多人模式提前做成当前功能。
- 禁止为了快速交付把核心状态写死在 View 层。
- 禁止无理由引入大型第三方状态管理或动画库。
