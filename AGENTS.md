# AGENTS.md

## 1. 作用
本文件用于约束 Codex 在项目「一起 / Toghter」中的执行方式。  
目标：减少偏航、降低返工、控制 token 消耗、保证代码结构稳定。

## 2. 响应原则
- 先给结论，再执行。
- 简单任务直接做；复杂任务先给“目标 / 影响范围 / 方案 / 风险”。
- 不确定时直接问清，不要猜。
- 不要客套、不要复述用户原话、不要输出空泛套话。
- 回复尽量短，但关键信息不能缺。

## 3. 首次进入项目时必须先读
- `DEVELOPMENT_GUIDELINES.md`
- 当前 `AGENTS.md`
- 最新 PRD / 页面需求 / 信息架构文档
- 与本次任务直接相关的 Features / Models / Services 文件

如果未读上述文件，不要直接开始大改代码。

## 4. 开发总原则
- 先把结构搭稳，再逐步开发页面。
- 优先修改现有文件，非必要不新建文件。
- 所有实现优先简单、清晰、可维护、可被 AI 理解。
- 不做过度抽象，不引入无必要复杂架构。
- 不允许把业务状态机直接散落在 View 里。
- 业务逻辑必须和 View 分离。
- 服务层先协议化，再实现具体服务。

## 5. 产品硬边界
- 这是双人协作 App，不是泛 Todo、泛社交、多人协作工具。
- 仅支持一对一绑定。
- 解绑后旧关系数据不得迁移给新对象；新绑定必须从零开始。
- 请求和待办本质上是同一类“事项”。
- 决策仅存在于决策域，只有部分决策可转事项。
- 任何改动不得擅自改变产品边界；若发现冲突，先指出再处理。

## 6. 技术与工程约束
- 平台：iPhone only。
- 技术：SwiftUI 为主；仅在必要时局部使用 UIKit。
- 当前阶段：本地数据 + mock 登录 + 可替换服务层。
- 长期后端方向：CloudKit；当前实现不得阻断未来接入。
- 页面必须支持 mock data 和 SwiftUI Preview。
- 开发时就考虑深色模式。
- 优先使用 Apple 原生 API 和系统能力，谨慎引入第三方库。

## 7. UI 实施规则
- 不要把项目做成普通任务工具的视觉语言。
- 风格方向：现代简洁、黑白粉、克制、轻生活感。
- 禁止廉价情侣风、过度粉色、过度装饰。
- 优先复用组件；非必要不允许页面内定义临时样式。
- 优先保证结构、状态、交互正确，再做像素级精修。

## 8. 执行顺序
默认按以下顺序推进：
1. 读文档与相关代码
2. 明确影响范围
3. 先模型与状态
4. 再服务与数据流
5. 再页面骨架
6. 再交互
7. 最后视觉精修与动效

不要一上来直接堆页面。

## 9. 复杂任务输出模板
复杂任务开始前先输出：
- 任务目标
- 影响模块
- 实现方案
- 风险与边界
- 需要确认的问题（如有）

完成后输出：
- 已完成内容
- 影响范围
- 未完成内容
- 下一步建议

## 10. 禁止事项
- 禁止跳过文档直接写代码。
- 禁止擅自新增功能。
- 禁止擅自修改数据规则、导航结构、核心对象定义。
- 禁止跨模块大改。
- 禁止创建临时脚本和无意义文件。
- 禁止为了省事把逻辑硬编码进单个页面。
- 禁止未说明理由就引入第三方库。
- 禁止生成与当前任务无关的大段样板代码。

## 11. 文件与命名规则
- 命名优先业务语义，不要泛化成 task / manager 式命名。
- 枚举、状态、常量集中定义，避免 magic number。
- 单页面拆为主视图 + 子组件。
- 单个 ViewModel 不承担多个业务域。
- Mock、Preview、Protocol 命名必须统一且可搜索。

## 12. 提交与修改策略
- 小步修改，小步验证。
- 一次提交只解决一个清晰问题。
- 修改前说明目的；修改后说明影响。
- 若任务只要求评估，不要直接改代码。
- 提交前保证项目可编译、关键 Preview 可运行。

## 13. 当存在不确定性时
遇到以下情况必须先问：
- 需求与文档冲突
- 影响核心边界
- 需要改动数据模型或状态机
- 需要新增第三方依赖
- 需要改变页面信息架构
- 需要删除或迁移用户数据

## 14. token 控制原则
- 回复优先短格式。
- 只展开与当前任务直接相关的信息。
- 不要重复描述已知背景。
- 不输出大段无关解释。
- 除非用户要求，不要一次性展开多个未来步骤。
